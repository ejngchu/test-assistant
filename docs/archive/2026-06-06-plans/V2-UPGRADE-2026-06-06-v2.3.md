## 一、产品定位

### 1.1 定位陈述

> **Venus-Mate 是一个面向全年龄段孩子的 AI 学习成长伴侣。** 孩子只需要拍照录入错题，系统就会自动推衍孩子的学习情况（能力图谱、知识盲点、学习节奏、进步轨迹），并针对性地进行辅导与帮助（同类练习、艾宾浩斯复习、知识薄弱点巩固）。

**核心特征：**
- 不分年级、不分地域 —— "三年级=加减法" 这种粗暴映射被取消
- 单入口操作 —— 拍错题是唯一的起点，分析、推衍、练习、复习全自动
- 数据驱动画像 —— 注册时填的"年级"只是一个粗标签，真正的画像从录入的第一道错题开始动态构建

### 1.2 目标用户

- **核心用户**：8-18 岁中小学生（小学~高中）
- **辅助用户**：家长（仅作为孩子数据的"消费者"——看报告、导出打印）
- **设计风格**：保留原有圆润、温暖、鼓励性基调（详见 `docs/DESIGN.md`），但文案/插画/色彩不再绑死"三年级"

### 1.3 核心价值主张

```
孩子做的事         系统自动完成
──────────       ────────────────────────────
📷 拍一道错题  →   🧠 识别科目 + 解析知识点 + 定位盲点
                →   📊 推衍学习画像（能力 / 节奏 / 趋势）
                →   📝 生成针对性练习
                →   📅 按艾宾浩斯曲线排复习
                →   📈 追踪掌握度变化，遗忘自动降权
                →   📤 数据可导出打印
```

**一句话：录一道错题，剩下的系统全自动。**

### 1.4 与 V1 的差异

| 维度 | V1（已归档 `docs/archive/PRD-v1.0-2026-06-02.md`）| V2 |
|------|------------------|-----|
| 目标用户 | 上海小学三年级 | **全年龄段中小学生**（8-18 岁） |
| 核心驱动 | 学习助手（多功能工具箱）| **错题驱动的学习成长伴侣** |
| 操作复杂度 | 多 TabBar 入口（首页/作业/错题本/练习/复习/设置） | **单入口：拍错题→全自动**（TabBar 精简为 3 个） |
| 数据策略 | Mock 数据展示 | **真实 AI 分析 + SQLite 持久化 + 智能遗忘** |
| 商业化 | 未定义 | 预留家长端（远期） + 微信支付（远期） |

## 二、技术栈：各端点作用与替代方案

### 2.1 全栈架构总览

```
┌──────────────────────────────────────────────┐
│  前端 (Taro 4 + React + Tailwind 4)          │  ←  小程序 / H5 / 抖音
│  渲染 UI、处理交互、调用 LLM 客户端          │
└────────────────┬─────────────────────────────┘
                 │ HTTPS (相对路径 /api/...)
                 ▼
┌──────────────────────────────────────────────┐
│  后端 (NestJS 10 + Drizzle ORM)              │  ←  Node.js 服务 (:3000)
│  RESTful API + LLM 调度 + 业务编排          │
└────┬─────────────┬──────────────┬────────────┘
     │             │              │
     ▼             ▼              ▼
  SQLite        LLM API       TOS / 本地磁盘
  (data.db)    (OpenAI compat) (对象存储/上传)
```

### 2.2 前端层

#### 2.2.1 Taro 4（React 渲染）

| 维度 | 说明 |
|------|------|
| **作用** | 把同一份 React 代码编译成微信小程序、H5、抖音小程序三端，**一套代码三端运行** |
| **满足需求？** | ✅ 满足。孩子在小程序里操作，开发者在 H5 调试，发布到多端 |
| **优势** | ✅ React 生态完整（Zustand/shadcn/ui/hooks）✅ 三端统一 ✅ Taro 4.x 已稳定 |
| **劣势** | ❌ 编译层抽象 → 调试报错位置不直观 ❌ 主包 ≤2MB（图片/字体必须外置）❌ H5 兼容性需手动调（`position:fixed+flex`）|
| **更轻量替代** | **Remax / Mor** —— 更轻量的 React→小程序方案；生态不成熟、组件库少 |
| **更原生替代** | **微信原生小程序（WXML/WXSS）** —— 性能最好；放弃跨端，3 套代码 |
| **结论** | **保留 Taro 4**。已有 9 个页面 + 完整组件库，迁移成本远大于收益 |

#### 2.2.2 Tailwind CSS v4 + weapp-tailwindcss

| 维度 | 说明 |
|------|------|
| **作用** | 原子化 CSS 框架 + 跨端适配层（小程序的类名 `bg-primary/90` 会被 `weapp-tailwindcss` 编译为合法语法）|
| **满足需求？** | ✅ 满足。设计系统、设计一致性、响应式全部依赖它 |
| **优势** | ✅ 开发效率极高 ✅ shadcn/ui 组件库依赖它 ✅ v4 Oxide 引擎更快 |
| **劣势** | ❌ weapp-tailwindcss 适配层有兼容性坑（已踩过 6 套方案才收敛）❌ 小程序不支持 `:hover`/`ring-*` |
| **更轻量替代** | **Less + BEM** —— 小程序原生友好；开发效率低、难约束设计系统 |
| **结论** | **保留**。`pnpm validate` 全绿，切换成本高于收益 |

#### 2.2.3 Zustand v5

| 维度 | 说明 |
|------|------|
| **作用** | 全局状态管理（`appStore.ts` 唯一数据源，承载所有领域类型）|
| **满足需求？** | ✅ 满足。9 个页面规模完全够用 |
| **优势** | ✅ 极轻量（<1KB gzipped）✅ API 简洁 ✅ 无 boilerplate |
| **劣势** | ❌ 大型应用缺中间件生态（对比 Redux Toolkit）|
| **更轻量替代** | **Jotai/Recoil** —— 原子化更细粒度；学习成本高 |
| **结论** | **保留**。当前规模最合适 |

#### 2.2.4 lucide-react-taro

| 维度 | 说明 |
|------|------|
| **作用** | 图标库（Taro 适配版，icon 编码为 `data:image/svg+xml` 通过 `<Image/>` 渲染）|
| **结论** | **保留**。不要用 `lucide-react`（原生版在小程序中无法运行）|

### 2.3 后端层

#### 2.3.1 NestJS 10

| 维度 | 说明 |
|------|------|
| **作用** | Node.js 服务端框架，提供 RESTful API（8 个端点）+ LLM 服务调度 + Module/Service 架构 |
| **满足需求？** | ⚠️ **过重但可用**。当前 8 个 API + 1 个 LLM service 不需要 NestJS 的全套 Module/Guard/Decorator |
| **优势** | ✅ 结构规范 ✅ DI 容器 ✅ 中间件生态完善 ✅ TypeScript first |
| **劣势** | ❌ 启动慢（~2-3s）❌ 学习曲线陡 ❌ 150+ 依赖 ❌ **8 个端点属于"防御性过度设计"** |
| **更轻量替代（强烈推荐备选）** | **Hono** —— ~14KB、Edge-native、TypeScript first、中间件模式同 Express。**1 天内可完成迁移**（Controller→Route、Service→Handler、DTO→Zod）|
| **更轻量替代 2** | **Fastify** —— 比 Express/NestJS 快 2-3x，插件丰富；仍比 Hono 重 |
| **更轻量替代 3** | **Express + 手写路由** —— 最轻量；缺结构、变大后混乱 |
| **结论** | **当前阶段保留 NestJS**（已有完整基础设施，迁移 ROI 低）。**退出机制**：如果开发/调试中 bug 过多，立即切 Hono |

#### 2.3.2 Drizzle ORM + better-sqlite3

| 维度 | 说明 |
|------|------|
| **作用** | 类型安全 ORM + SQLite 文件数据库（`server/data.db`），零外部服务 |
| **满足需求？** | ✅ **完美匹配**。单用户场景、轻量、本地文件、无需安装 MySQL/PostgreSQL |
| **优势** | ✅ TypeScript 类型推导 ✅ 极轻量（比 Prisma 小 10x）✅ SQLite 一流支持 ✅ Schema 即 migration source |
| **劣势** | ❌ 生态较新（vs Prisma）❌ 无 GUI（vs Prisma Studio）|
| **更轻量替代** | **原始 better-sqlite3** —— 最轻量；手写 SQL 易出错 |
| **更重替代** | **Prisma** —— GUI 好；bundle 较大，对 SQLite 需额外配置 |
| **结论** | **Drizzle + SQLite 是当前最佳组合**。schema/migration/provider 三件套已就位 |

### 2.4 LLM 层

| 维度 | 说明 |
|------|------|
| **作用** | 调用大模型做 OCR（题目识别）+ 文本分析（知识点/盲点）+ 题目生成（同类练习）|
| **当前实现** | `server/src/common/services/llm.service.ts` 走 **OpenAI 兼容接口**（`/v1/chat/completions`）|
| **满足需求？** | ✅ 满足。Vision 模型支持图片输入、JSON 模式可强制结构化输出 |
| **推荐 Provider** | 见 §5 P-01 —— **优先 SiliconFlow（免费 + 多模型）**，备选智谱 GLM-4-Flash / 通义千问 |
| **结论** | **保持 OpenAI 兼容接口**，不绑死任何一家 provider |

### 2.5 各端点轻量化评估

| 端点 | 复杂度 | 当前实现 | 是否有更轻量替代 |
|------|--------|----------|------------------|
| `POST /study/subject/detect` | 低 | LLM 调一次 | 可降级为本地启发式（颜色/文字匹配），但 LLM 准 |
| `POST /study/homework/check` | 中 | LLM 批量识别 | 无 |
| `POST /study/mistake/analyze` | 低 | LLM 调一次 + DB 写 | 无 |
| `GET /study/review/plan` | 低 | 纯算法（艾宾浩斯）+ DB 读 | 无（已经是后端最便宜的）|
| `POST /study/review/complete` | 低 | DB 更新 | 无 |
| `POST /study/practice/generate` | 中 | LLM 生成 + DB 写 | 无 |
| `GET /study/practice/history` | 低 | DB 读 | 无 |
| `POST /study/practice/submit` | 低 | 答案比对 + DB 写 | 无 |
| `POST /settings/mock-mode` | 低 | 内存 flag | 无 |
| `POST /upload` | 中 | multer + 本地磁盘 | 后续接 TOS（见 P-04）|

**结论：8 个业务端点本身复杂度低，NestJS 的分层对当前规模属于"防御性过度设计"，但不影响功能正确性。** 切换到 Hono 1 天内可完成，ROI 由"开发体验是否拖慢进度"决定。

### 2.6 技术栈总结

| 层级 | 选型 | 是否满足 | 操作 |
|------|------|----------|------|
| 前端框架 | Taro 4 + React 18 | ✅ | 保留 |
| 样式 | Tailwind 4 + weapp-tailwindcss | ✅ | 保留 |
| 状态 | Zustand 5 | ✅ | 保留 |
| 图标 | lucide-react-taro | ✅ | 保留 |
| 后端框架 | NestJS 10 | ⚠️ 过重但可用 | **保留**（退出机制：bug 过多→Hono） |
| ORM | Drizzle + better-sqlite3 | ✅ | 保留 + 接入 |
| LLM | OpenAI 兼容接口（SiliconFlow 等）| ✅ | 切换 Provider |
| 对象存储 | 本地磁盘（→ TOS）| ⚠️ 临时 | 保留 + 后续接 TOS |

## 三、Mock 演示如何顺利运行

### 3.1 端点选择：优先微信小程序，H5 是开发辅助

**结论：演示目标端点是微信小程序。** H5 是开发者日常调试工具，最终用户体验在小程序。

| 端点 | 角色 | 推荐度 | 说明 |
|------|------|--------|------|
| **微信小程序** | **演示目标** | ⭐⭐⭐⭐⭐ | 用户最终体验；优先保证跑通 |
| H5 | 开发辅助 | ⭐⭐⭐ | 浏览器快速验证，CI/单元测试友好 |
| 抖音小程序 | 二期扩展 | ⭐ | 暂不重点 |

### 3.2 Mock 数据流程

```
┌──────────────────────────────────────────────────┐
│  后端启动，读取 .env.local                         │
│  LLM_PROVIDER=openai 但 LLM_MOCK_MODE=true       │
└────────────────┬─────────────────────────────────┘
                 │
                 ▼
┌──────────────────────────────────────────────────┐
│  llm.service.ts：mock 模式 = 返回固定结构化假数据 │
│  - subject/detect → 随机 chinese/math/english    │
│  - mistake/analyze → 返回"有余数除法"分析        │
│  - practice/generate → 从 mock 题库抽 5 道       │
│  - 所有 review/practice 走真实 DB（SQLite）      │
└────────────────┬─────────────────────────────────┘
                 │
                 ▼
┌──────────────────────────────────────────────────┐
│  前端调 /api/study/* 返回 200 + {code, data}     │
│  Network 不解包，res.data.data 是业务数据         │
└──────────────────────────────────────────────────┘
```

**关键点：**
- Mock 模式**不需要任何外部 API Key**（已默认开启）
- Mock 数据**走真实 SQLite**，可以真实 CRUD、可以验艾宾浩斯算法
- H5/小程序**用同一份后端**，演示路径完全一致
- **Settings 页面有"演示模式"开关** —— `POST /api/settings/mock-mode` 运行时切换 mock/真实 LLM，无需重启

### 3.3 演示前必做

```bash
# 1. 安装依赖
pnpm install

# 2. 启动 SQLite（自动创建 data.db）
cd server && DATABASE_URL=./data.db pnpm db:push   # 首次执行迁移
cd .. && pnpm dev:server                            # 启动后端

# 3. 启动前端（两个终端任选）
pnpm dev:web    # H5 → http://localhost:5000/index.html
pnpm dev:weapp  # 小程序 watch → dist/，用微信开发者工具导入

# 4. 打开 Settings 页面，确认"演示模式"开关已打开
```

### 3.4 端到端演示路径

```
步骤 1: 首页
  → 看问候语 + 2×2 统计（待复习/错题数/正确率/连续天数）
  → TabBar 5 个（首页/作业/错题本/练习/我的）

步骤 2: 错题本 Tab
  → 3 条 mock 错题（自动 mock 模式）
  → 右上角 +  → 拍照/选图 → 详情页（自动调 /api/study/mistake/analyze，mock 模式返回"有余数除法"）

步骤 3: 作业 Tab
  → 拍照 → /api/study/subject/detect（mock 随机返回科目）

步骤 4: 练习 Tab

```

### 3.5 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `pnpm dev:web` 报 `ERR_INVALID_ARG_VALUE` | Taro 4 Vite 编译问题 | 确认 `config/index.ts` 未加 `h5.postcss.tailwindcss` |

## 四、为什么项目与 Coze 有关 —— 不是 Coze 平台，但由 Coze CLI 脚手架创建

### 4.1 简短回答

- **不是**字节跳动的 Coze（扣子）平台本身 —— 我们**不依赖** Coze 的 Bot/工作流/插件
- **是**字节跳动 Coze CLI 脚手架生成了项目骨架 —— `package.json#name` 至今还是 `coze-mini-program`
- 后续 LLM 已切换到 **OpenAI 兼容接口**（多 Provider），与 Coze 平台解耦

### 4.2 项目为什么带 "coze" 字样

| 时间 | 状态 | 说明 |
|------|------|------|
| 初始 | Coze CLI 创建 | 字节跳动 Coze 平台的 CLI 工具能直接生成 Taro + NestJS 全栈项目模板，本项目最初通过它脚手架 |
| 当前 | 已脱离 Coze 平台 | 业务代码、API 设计、LLM 接入都不依赖 Coze；只保留 `coze-mini-program` 作为 package name（package name 改名需要改很多地方，无实际收益）|
| 远期 | 完全独立 | 如需彻底解绑，把 package name 改为 `venus-mate`，可一次 PR 完成 |

### 4.3 Coze 平台 vs 当前方案

| 方案 | 优势 | 劣势 | 是否采用 |
|------|------|------|----------|
| **Coze 平台**（扣子）| ✅ Bot/工作流/插件/知识库可视化 ✅ 不需要写 LLM 调度代码 ✅ 字节系生态 | ❌ 受 Coze 平台限制 ❌ 计费依赖 Coze ❌ 私有化部署困难 ❌ 部分高级能力收费 | ❌ **不采用** |
| **OpenAI 兼容接口**（多 Provider）| ✅ 自由切换 provider ✅ 私有化部署简单 ✅ 不被任何平台绑定 ✅ 单次 API 调用 | ❌ 需自己写 prompt 工程 | ✅ **采用** |
| **本地模型**（Ollama/vLLM）| ✅ 完全离线 ✅ 数据不出本地 | ❌ 硬件要求高 ❌ 模型质量参差 | ⏳ 远期考虑 |
| **自训练模型** | ✅ 极致定制 | ❌ 数据/算力成本巨大 | ❌ 不考虑 |

### 4.4 替代方案详解

如果未来想**完全脱离** Coze CLI 脚手架的"印记"，可以：

1. **改 package name**：`coze-mini-program` → `venus-mate`（一次性操作，CI/部署配置要同步改）
2. **移除 Coze 相关代码**：`llm.service.ts` 中的 `callCozeAPI()` 方法可以删除（保留也无害，默认不调用）
3. **迁移到更轻量框架**：见 §2.3.1，NestJS → Hono 1 天内可完成

**当前结论：保留 Coze CLI 脚手架遗产，不主动剥离。** 项目已经实质脱离 Coze 平台，剩余的"coze"字样只是历史包袱，不影响功能。

#### 5.2.1 T-03 复习提醒推送实现方案

**采用：方案 A（微信订阅消息）+ 方案 C（小程序内红点）组合**

| 方案 | 描述 | 优势 | 劣势 | 状态 |
|------|------|------|------|------|
| **A. 微信订阅消息** | 用户首次点击"开启复习提醒"→ `wx.requestSubscribeMessage` 弹窗授权 → 后端调用 `subscribeMessage.send` 推模板消息 | ✅ 用户感知最好 ✅ 到达率高 ✅ 官方稳定 | ❌ 需主动授权 ❌ 模板内容固定（需申请）❌ 单次订阅限发 1 条 | ✅ 采用 |
| **B. 微信服务通知** | 不受订阅次数限制 | — | ❌ 教育类难批 | ❌ 不推荐 |
| **C. 小程序内红点** | 后端定时计算 + 前端进入时展示红点 | ✅ 无需审核 ✅ 实现简单 | ❌ 用户不看就不知道 | ✅ 采用（兜底）|
| D. 第三方推送 | 极光/个推 | 灵活 | ❌ 小程序环境受限 | ❌ 不推荐 |

**实现要点：**

```
复习时间到达
  ↓
后端：cron job 每天 09:00 跑 review/reminders 任务
  ↓
  ├─ 已有授权用户 → 调用 subscribeMessage.send(OPENID, TEMPLATE_ID, data)
  └─ 未授权用户  → 写入 in_app_notifications 表
  ↓
前端：进入首页时拉 /api/study/review/in-app-notifications
  → 显示红点 + 未读列表
```

**为什么不用 B/D：** 教育类小程序难批服务通知；第三方推送在小程序中受限。
